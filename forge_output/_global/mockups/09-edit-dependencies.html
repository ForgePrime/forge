📊 Edit impact

If you removed O-001:
  • O-002 stays ACTIVE (still has O-006 hard-dep)
  • Future analysis tasks lose KB scope inheritance from O-001
  • The original "why this objective exists" rationale loses its anchor
If you converted O-006 to soft-dep:
  • O-002 could start tasks before O-006 ACHIEVED
  • BUT planning would have to assume cluster details — risk of rework if O-006 lands differently
  • O-005 / O-007 timing may shift earlier (cascade)
🔎

Dependency edits have second-order effects

  • Adding a hard-dep locks O-002 until N other objectives ACHIEVE. If those drift, O-002 drifts too. The new dep doesn't introduce work, it introduces wait.
  • Removing a dep may expose unresolved assumptions. If O-002's planning quietly assumed O-006's cluster version, removing the dep doesn't remove the assumption — it just hides it from the DAG. Add a note.
  • Soft-dep means "preferred order, not required." The challenger does NOT enforce soft-deps at completion time. They're advisory for the planner.
  • Cycle prevention is structural, not semantic. Two objectives can have circular reasoning ("A justifies B; B justifies A") without forming a graph cycle. Forge does not catch that.
  • Blocks-list is computed. You can't edit it from here on purpose — to keep the source-of-truth single. Edit downstream objectives' dependencies directly.
  • No notification to dependent owners. If you remove O-001's link, O-001's owner (hergati) is not notified. Communicate manually if it matters.

09-edit-dependencies · data sources

  1. Header counts ← objective.depends_on[] length + computed objective.blocks[] (reverse-index of all other objectives' depends_on).
  2. DAG visualization ← deterministic graph render. Nodes = objectives, edges = deps. O-002 is highlighted via URL ?focus=O-002. Layout: 3 columns (upstream, focus, downstream).
  3. Depends-on rows (editable) ← objective.depends_on[] JSONB array. Each entry {objective_id, kind: 'hard'|'soft'}. Inline toggle = PATCH. Remove = PATCH.
  4. Add dependency dropdown ← query objective WHERE project_id=acme-erp-migration AND id != O-002. Each option pre-validated for cycles client-side using cached graph; cycles disabled w/ tooltip showing the forming path.
  5. Blocks list (read-only) ← reverse-query objective.depends_on[] across all project objectives. Each row links to the downstream objective's edit-deps page.
  6. Impact preview ← deterministic graph walk. "If removed X" computes which objectives lose blocking, which KRs change status. Currently shows hypothetical for the 2 existing deps.
  7. Scrutiny — second-order effects ← enforced template. Always shown.
Destinations: Each edit → PATCH /objective/{id}/dependencies. Remove with downstream impact → may pop 16-preview-apply-modal. "View full DAG" → 03v2-objectives-dag.html. "edit X deps" links → same page w/ different focus.

Note: existing schema already has objective.depends_on[]. The hard/soft kind toggle implies that array stores objects not just ids — confirm schema supports JSONB shape.
