🔎 What phase defaults do NOT override
  • Skills with auto-attach rules that match — those still fire regardless of phase defaults
  • Skills attached manually on a specific task — task-level overrides phase-level
  • Post-stage hooks — see separate hooks tab
  • Challenger's own skill pool — configured separately on the op contract tab
In other words: these are the "if nothing else fires, run these" defaults. Order of precedence: task-level > auto-attach rule > phase defaults > org defaults.

12-phase-defaults-tab · data sources

  1. Scrutiny ← static. Teaches precedence order.
  2. 4 columns = enum phase = [ANALYSIS, PLANNING, DEVELOP, DOCUMENTATION]. Each column lists skills where phase_default.phase = X AND phase_default.project_id = acme-erp-migration.
  3. Customized badge on DEVELOP column = project_phase_default.overrides_org_default = true. Shown only when project has edited defaults for that phase.
  4. "Added for this project" pill ← phase_default.source = 'project' (vs 'org'). Org-level ones can't be removed, only overridden.
  5. Cost overhead = SUM(skill.cost_impact_per_call) for the skills in that column.
Destinations: "+ Add skill to X defaults" → modal picker listing skills not yet in that phase's defaults. "×" on a skill → POST /project/acme-erp-migration/phase-defaults/{phase}/remove/{skill_id}. "Reset to org defaults" → confirmation modal → POST /project/acme-erp-migration/phase-defaults/reset. Save → POST bulk update.
