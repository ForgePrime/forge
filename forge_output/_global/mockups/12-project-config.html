# Operational contract — acme-erp-migration
# This file is part of every prompt Forge sends to the LLM.
# Rules here override defaults but not hard gates.

## Hard constraints (from SRC-004 client manual note)
- MUST be on-prem deployable · no external SaaS
- MUST be Python stack · team has 4y Django experience
- MUST NOT introduce Oracle dependency
- MUST meet HIPAA minimum (session + audit log)
- Budget: 2 backend + 1 frontend devs · 4 months

## Preferred stack
- Django 5 + DRF · PostgreSQL 16 · Redis (session)
- pytest · ruff · mypy strict on app/
- Alembic migrations · tests for up + down

## Patterns to follow
- Strangler-fig migration (per ADR-001)
- Audit log every patient data read/write (HIPAA)
- Avoid premature abstractions · 3 similar blocks < 1 bad abstraction

## Patterns to avoid
- No raw SQL unless explicitly justified
- No singleton patterns for DB / cache
- No "magic" auto-register · explicit wiring
- No hidden network calls during migration scripts

## Tone / style
- Production-minded, not tutorial-minded
- Comments only where non-obvious (mechanical reviewers reject comment noise)
- Pytest test names describe failure (e.g., test_returns_404_when_user_not_found)

## Review obligations
- Every dev task must cite SRC or objective reference in commit message
- Every challenger must note its scope limits explicitly
- Every finding dismissal must include >= 50 chars reason

12. Project config — annotations

  1. Changes apply to FUTURE tasks only. Past tasks keep their original config — audit trail preserved. Prevents retroactive rewrites.
  2. Operational contract is editable, versioned. Every revision logged. LLM can suggest improvements (read-only suggestion, user approves). Injected as ~420 tokens into every prompt.
  3. Active skills sidebar — at-a-glance audit of what's in every prompt. Cost overhead estimate visible. "Rebalance" link suggests trims if cost creeps.
  4. Post-stage hooks. Fire AFTER a stage regardless of task config. Project-wide invariants. 4 types: after analysis / planning / develop / doc.
  5. Autonomy roadmap (L1–L5). Progressive hand-off. Each level requires audit pass of previous. Concrete blockers shown ("L3 requires clean audit — 3 pre-selected-ignored ambiguities"). Trust earned incrementally.
Key: Config is opinion + evidence, not just toggles. User sees WHY each rule exists (cost, audit, constraint) and WHAT it blocks.
