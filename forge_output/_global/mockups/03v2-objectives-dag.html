Task types explained — 4 types, same structure

ANALYSIS Business analyst role

Reads KB sources, answers questions, finds ambiguities, writes objectives and KR definitions. Output: objectives + decisions.

AC examples: "Every business rule from SRC-001 has been mapped to a KR" · "All unresolvable questions are logged as OPEN decisions"
PLANNING Tech lead role

Takes an objective + KR, decomposes into executable develop tasks with AC and dependencies. Output: task DAG.

AC examples: "Every KR has ≥1 task that measurably achieves it" · "No task has >5 ACs" · "Task DAG is acyclic"
DEVELOP Engineer role

Writes code, migrations, tests. Runs Phase A/B/C verification. Output: merged diff + passing tests + findings.

AC examples: "pytest tests/test_x.py::test_y passes" · "migration upgrades + downgrades run cleanly"
DOC Tech writer role

Writes architecture decision records, API reference, user guide, runbook. Uses deliverables + code as source. Output: .md files.

AC examples: "Every public endpoint has API docs" · "ADR captures the on-prem rationale" · "User guide covers 5 primary workflows"
All 4 types share: acceptance criteria · gates · test scenarios · challenger review · re-openable status · DAG dependency · Direct / Prompt-crafted mode
